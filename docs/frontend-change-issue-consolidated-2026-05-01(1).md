## 1. 本次已落地后端变更（可直接联调）

### 1.1 手机号默认脱敏 + 按钮查看完整号

1. 客户列表/详情 `phone` 默认返回脱敏值（如 `139****0001`）。
2. 新增查看完整号接口：
   - `GET /api/v1/admin/customers/{uid}/phone`
   - `GET /api/v1/staff/customers/{uid}/phone`
3. 每次成功查看会写审计表 `customer_phone_view_log`（仅存脱敏号，不存明文）。

返回字段（查看接口）：
1. `uid`
2. `phone`（完整手机号）
3. `maskedPhone`
4. `viewedAt`

### 1.2 标签衰减常数 + 标签衰减度（已落地）

1. 新增独立配置表 `tag_decay_config`（与行为事件 `decay_config` 解耦）。
2. 管理员衰减配置接口支持虚拟项 `TAG_DECAY`：
   - `GET /api/v1/admin/decay-config` 返回包含 `TAG_DECAY`
   - `PUT /api/v1/admin/decay-config/TAG_DECAY` 可更新标签衰减常数
3. 用户标签列表新增字段：
   - `decayPercent`（0~100，保留 1 位小数）
4. 口径：
   - 按 `user_tag.created_at` 到当前时间计算
   - 公式：`Wt = W_init * e^(-λt)`

接口影响：
1. `GET /api/v1/admin/customers/{uid}/tags`
2. `GET /api/v1/staff/customers/{uid}/tags`

## 2. 历史问题（未修，需前端按此执行）

### 2.1 P0：真机 AI 流式异常（会话可复用但无增量渲染）

当前状态：未修（需真机闭环验证）

现象：
1. 真机日志出现 `event start` 与 `event done`，但中间无稳定 `delta`。
2. 后端日志/Redis 有完整回复，但前端气泡不展示增量文本。
3. 部分端报过 `TextDecoder is not defined`，说明运行时兼容不一致。

影响：
1. AI 专家核心体验不可用或不稳定，属 P0。
2. 用户感知为“AI 不回复”，直接影响咨询链路。

根因（前端侧）：
1. SSE chunk 解码兼容不足，存在机型差异。
2. UTF-8 多字节跨 chunk 半包处理不稳。
3. `delta` payload 解析假设过强，未覆盖全部格式。
4. `done` 后缺少可靠回填兜底。

整改要求：
1. 统一 SSE 解码策略，移除对单一运行时能力的硬依赖。
2. 增加半包缓存与尾包 flush 机制。
3. `delta` 解析兼容对象/字符串两类结构。
4. `done` 后若气泡为空，自动走历史接口回填最近 assistant 消息。

验收：
1. 真机稳定出现 `start -> delta(>=1) -> done`。
2. 即使 `delta` 丢失，`done` 后可自动回填可见回复。
3. 同账号会话可持续复用，前端可见输出稳定。

### 2.2 P0：退出 AI 页重进后消息倍增

当前状态：未修

现象：
1. 真机更容易出现历史消息 2 倍或多倍重复。
2. 同一时段可能有重复历史查询，前端最终叠加渲染。

影响：
1. 聊天窗口污染，严重影响可读性和用户信任。
2. 后续分析/埋点也可能被重复展示干扰。

根因（前端侧）：
1. 退出时在途请求未失效，返回后仍写入页面状态。
2. 重进后二次初始化请求与旧请求结果叠加。
3. 历史消息合并缺少强去重键。

整改要求：
1. 历史查询增加请求版本号，仅消费当前版本返回。
2. 页面销毁时使在途请求结果失效。
3. 页面重进时重置首屏状态并保证“单次进入仅初始化一次”。
4. 合并时按 `messageId/seqNo+role` 去重。

验收：
1. 连续 3 次进出页面，历史条数稳定不倍增。
2. 上拉分页不出现重复条目。

### 2.3 P0：退出登录误清理 AI 会话，重登后无法恢复

当前状态：未修

现象：
1. 同账号登出再登录后进入 AI 页，会话断裂或新建。
2. 后端有历史消息，但前端无法恢复到原会话上下文。

影响：
1. 对话连续性中断，用户体验显著下降。
2. 模型上下文丢失，回复质量波动。

根因（前端侧）：
1. 登出执行了全量本地清理，误删 `uid -> sessionId` 映射。
2. 登录恢复流程未优先尝试本地/后端会话恢复。

整改要求：
1. 登出只清理鉴权敏感信息，不清理 AI 会话映射。
2. 会话键改为按用户隔离（如 `ai_session:{uid}`）。
3. 登录后进入 AI 页先恢复会话，再决定是否新建。
4. 账号切换仅清理旧账号对应键，不影响其他账号。

验收：
1. 同账号“登录->对话->登出->再登录”可继续原会话。
2. 切换账号不会串会话。

### 2.4 P1：管理端 Banner 管理未调用真实接口

当前状态：未修

现象：
1. 页面看似可新增/编辑/上下线/删除，但网络无真实 API 调用或未全链路落库。
2. 刷新后状态丢失或与后端数据不一致。

影响：
1. 无法完成管理端内容联调与验收。

当前联调主口径（员工端）：
1. `GET /api/v1/staff/content/banners`
2. `GET /api/v1/staff/content/banners/{id}`
3. `POST /api/v1/staff/content/banners`
4. `PUT /api/v1/staff/content/banners/{id}`
5. `DELETE /api/v1/staff/content/banners/{id}`

兼容/历史口径（不作为前端主调用）：
1. `GET /api/v1/admin/content/banners`
2. `GET /api/v1/admin/content/banners/{id}`
3. `POST /api/v1/admin/content/banners`
4. `PUT /api/v1/admin/content/banners/{id}`
5. `DELETE /api/v1/admin/content/banners/{id}`

前端整改要求：
1. 页面 CRUD 全部切到真实 API。
2. 写操作成功后强制刷新列表，避免本地状态漂移。

验收：
1. Network 可见完整 CRUD 请求。
2. 刷新后数据与后端一致。

### 2.6 P1：C 端套房页已登录仍显示登录拦截 + 占位异常

当前状态：未修

现象：
1. 已登录用户仍看到“登录后查看套房详情”。
2. 列表出现灰色伪卡片占位，导致内容断层。

影响：
1. 登录态与页面状态冲突，影响转化链路。

根因（前端侧）：
1. 登录态来源不统一。
2. 拦截模块与正常内容未做互斥渲染。
3. 占位数据与真实数据混渲。

整改要求：
1. 登录态统一走全局鉴权状态。
2. 套房内容与登录拦截改为严格互斥。
3. 去掉伪卡片占位，改为空态组件。

验收：
1. 已登录用户不再看到登录拦截。
2. 列表无灰色伪卡片。

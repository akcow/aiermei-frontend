# 前端交接文档

更新时间：2026-04-15

## 当前状态

1. 当前不依赖鉴权的接口整体基本可用，可用于页面主流程联调。  
2. 当前待解决问题集中在 AI 专家模块（会话连续性、等待态、SSE 丢包）。

## 当前后端鉴权方案

1. 受保护接口统一使用 `Authorization: Bearer <token>`。  
2. 后端鉴权链路为：校验 JWT -> 校验 Redis 会话 -> 写入请求上下文 `CURRENT_UID`。  
3. AI 相关接口（`/api/v1/ai/chat`、`/api/v1/ai/sessions/{sessionId}/messages`）已改为只从鉴权上下文取用户，不再信任 `X-Uid`。  
4. token 无效或过期时，后端返回未授权（业务码 `4003`）。  
5. 当前工程里仍保留请求头 UID 的兼容读取实现，用于旧链路过渡；前端新代码请不要再传 `X-Uid`。

## 微信接入修改建议

1. 请求层统一注入 Bearer（普通请求 + SSE 请求保持一致）。  
2. 删除 AI 页面手动传 `X-Uid` 的逻辑，避免与 token 鉴权混用。  
3. 收到 `4003` 后统一清理本地登录态并引导重新登录。  
4. 同步更新前端接口文档/OpenAPI，移除 `X-Uid` 请求头描述。  
5. 联调时先登录拿 token，再调用 AI 接口，避免匿名态误判为接口故障。

## 问题 1：同一用户每次进入 AI 专家都会新开 session

### 现象

1. 同一用户每次重新进入 AI 专家页面后再次提问，会开启新的 `sessionId`。  
2. 之前对话无法在前端连续展示，历史上下文割裂。

### 目标行为

1. 同一用户在有效期内复用同一个会话（或最近活跃会话）。  
2. 默认拉取最近 10 轮对话（10 对 Q/A）用于初始化展示。  
3. 聊天上滑时继续分页加载更早记录。

### 前端推荐修改方案

1. 持久化 `sessionId`（按 `uid` 维度本地存储）。  
2. 进入 AI 专家时优先恢复 `sessionId`，无则走新会话。  
3. 首屏加载最近 10 对话；滚动到顶部触发“上滑加载更多”。

### 后端新增接口

`GET /api/v1/ai/sessions/{sessionId}/messages`

请求参数：

1. 鉴权：`Authorization: Bearer <token>`（必须）。  
2. `cursor`（可选）：上一页最早消息的游标（建议使用 `seqNo` 或时间戳）。  
3. `limit`（可选）：默认 `20`，最大 `50`。  
   说明：若按“10 对话”口径，建议首屏请求 `limit=20`（用户/助手各一条）。

说明：`/api/v1/ai/chat` 与历史消息接口都已改为从 token 鉴权上下文读取用户，不再使用 `X-Uid`。

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "sessionId": "chat_19100001",
    "list": [
      {
        "seqNo": 101,

        "createdAt": "2026-04-15T13:10:00+08:00"
      },

        "seqNo": 102,

```

## 问题 2：AI 等待模型响应时缺少等待动画

### 现象

1. 用户发送消息后，若首个 `delta` 返回较慢，界面无明显“正在思考”反馈。  
2. 用户容易误判为请求失败或卡死。

### 前端推荐修改方案

1. 发送后立即展示 AI 占位气泡 + 加载动效（如三点跳动）。  
2. 收到首个 `delta` 后替换为真实内容流式追加。  
3. 收到 `done/error` 后关闭等待动画。

2. 后端已落库/缓存有内容，但前端气泡为空或缺字。

### 原因（前端侧）

1. SSE 解析未处理跨 chunk 半包/粘包。  
2. 渲染层对 `event.data` 结构假设过强（仅按 `event.data.content` 读取）。

### 前端推荐修改方案

1. 解析器增加 buffer，按 SSE 帧分割（`\n\n`）后再解析事件。  
2. 兼容 `event.data` 为对象与字符串两种情况。  
3. 对异常帧做容错，不应中断整条流。

## 问题 4：AI 专家页面缺少返回键

### 现象

1. 用户进入 AI 专家全屏页后，缺少明显的返回入口。  
2. 只能依赖非常规关闭操作，影响可用性。

### 前端推荐方案

1. 在 AI 顶部导航区提供明确的返回按钮（左上角箭头）。  
2. 返回行为统一：优先返回上一页；若无上一页则关闭抽屉/回首页。  
3. 返回按钮需覆盖 iOS/Android/小程序安全区场景。

## 问题 5：Markdown 渲染异常

### 现象

1. AI 回复中的 Markdown 语法出现原样文本（如 `**`、列表符号错乱）。  
2. 段落、换行、列表在不同消息上表现不一致。

### 前端推荐方案

1. 增加统一 Markdown 渲染链路（建议：先标准化，再渲染）。  
2. 支持基础语法：标题、加粗、列表、换行、代码块。  
3. 渲染前做安全处理（XSS 过滤/白名单），避免直接注入原始 HTML。  
4. 对异常 Markdown 文本提供降级展示（纯文本回退），不影响阅读。

## 问题 7：会员页未调用 `/users/me`

### 现象

1. 进入会员页时，前端未请求 `GET /api/v1/users/me`。  
2. 会员页展示主要依赖本地缓存状态，未与后端用户态实时对齐。

### 影响

1. token 过期、服务端资料更新后，页面可能仍展示旧状态。  
2. 已登录/未登录状态与服务端真实状态可能不一致。

### 前端推荐方案

1. 在会员页 `onShow` 阶段调用 `GET /api/v1/users/me`。  
2. 成功后回填本地用户资料与登录态；失败 `4003` 时清理本地会话并引导登录。  
3. 保留本地缓存作为兜底，但以后端返回为准。

## 问题 8：用户行为记录缺少前后端闭环

### 现状

1. 前端当前主要记录本地 `trackPath`，未形成稳定后端上报闭环。  
2. 管理端存在用户旅程查询需求，但后端接口未完整落地。

### 前后端约定（v1）

1. Base URL：`/api/v1`。  
2. 鉴权：统一 `Authorization: Bearer <token>`，不使用 `X-Uid`。  
3. 事件上报接口：`POST /api/v1/analytics/events/batch`。  
4. 用户旅程接口：`GET /api/v1/analytics/users/{uid}/journey?limit=100`。  
5. 上报事件字段约定：
   - `eventId`（必填，幂等去重）
   - `eventType`（必填：`PAGE_VIEW/CLICK/TAB_CHANGE/SUBMIT/LOGIN/APPOINTMENT_INTENT`）
   - `path`（必填）
   - `pathName`（可选）
   - `occurredAt`（必填，ISO8601 带时区）
   - `durationSeconds`（可选）
   - `metadata`（可选 JSON）
6. 前端发送策略：
   - 本地队列缓存埋点；
   - 页面隐藏/应用后台/定时批量 flush；
   - 单批建议不超过 20 条；
   - 失败最多重试 3 次，指数退避；
   - 收到 `4003` 立即停止上报并触发登录态恢复。
7. 后端处理策略：
   - 按 `eventId` 幂等；
   - 参数错误返回 `4000`；
   - 未授权返回 `4003`；
   - 成功返回 `code=0`。

## 验收口径（仅本次范围）

1. 同一用户反复进入 AI 专家，不再频繁新建 session。  
2. 首屏能拿到最近 10 对话，上滑可继续加载历史。  
3. 发送问题后有等待动画，首包到达后平滑切换。  
4. “你好”等短文本在 SSE 下稳定展示，不再出现丢字/空白。
5. AI 专家页面具备明确返回键，真机操作路径一致。
6. Markdown 展示稳定，无明显 `**` 原样泄露和段落错乱。
7. 会员页进入时会请求 `/users/me`，并能正确处理 `4003` 失效态。
8. 行为埋点可批量上报并可查询用户旅程，形成前后端闭环。
